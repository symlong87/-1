# 餐厅预订页面 P2 优化指导 v2

## 文档目标

这份文档基于以下两个文件重新整理：

- `餐厅预订-优化版-v2.html`
- `餐厅预订-P2优化指导-v1.md`

这版文档的目标不是重复旧版 `P2` 提纲，而是结合当前 `v2` 页面现状，重新回答三个问题：

1. 旧版 `P2` 中哪些内容其实已经在 `v2` 页面落地
2. 哪些旧建议如果继续照搬，会造成重复优化
3. 在当前完成度下，下一轮真正还值得做的 `P2` 应该是什么

结合 `v2` 页面来看，这一版页面已经不只是“P1 完成版”，而是已经提前覆盖了不少原始 `P2` 能力，包括：

- 日期卡片上的库存趋势提示
- 时间状态分层表达
- 人数快捷选择与场景化提示
- 备注标签化
- 吸底确认区联动
- 成功反馈与异常兜底

因此，新版 `P2` 的重点不应再是“把这些能力从 0 做出来”，而应转向：

1. 让不可订场景有更明确的替代路径
2. 让人数和需求真正影响推荐逻辑
3. 让提交后的状态更像真实产品流程
4. 让整页状态表达更一致、更可信

## 一、对照 v2 后的重复项检查结论

以下内容在旧版 `餐厅预订-P2优化指导-v1.md` 中属于重点，但结合 `餐厅预订-优化版-v2.html` 来看，已经不适合作为下一轮 P2 的主任务。

## 1. 日期库存表达精细化

- `v2` 已经在日期卡片中加入了趋势型提示，例如：
  - `午市可订充足`
  - `晚市更容易订到`
  - `晚市高峰较紧张`
  - `可订但大桌有限`
- 时间区也已经用 `可订 / 推荐 / 紧张 / 已满` 做了更细分的时段状态表达
- 日期切换后会联动刷新时段，不再是静态表单字段

结论：

- 不再把“补上日期库存表达”作为本轮 P2 主任务
- 下一轮要处理的是“状态怎么驱动后续决策”，而不是“再多写几个状态词”

## 2. 人数与场景引导增强

- `v2` 已经提供快捷人数选择
- 已经根据人数变化动态更新提示文案
- 已经在首屏和备注区对聚餐、包间、安静位置、带娃等场景做了引导

结论：

- 不再把“新增人数提示”作为本轮 P2 重点
- 下一轮应进一步处理“人数是否真正影响推荐结果”

## 3. 提交反馈增强

- `v2` 已经具备提交后的 toast 成功反馈
- 已经有页面内成功卡片
- 已经模拟了提交中状态和提交成功后的结果说明
- 已经补上基础异常提示和高峰时段兜底拦截

结论：

- 不再把“加一个成功提示”视为 P2 任务
- 下一轮重点应是“提交后还能做什么”，而不是“提交后有没有提示”

## 4. 交互细节打磨

- `v2` 已经有比较完整的选中态、禁用态、错误态和吸底区样式
- 吸底确认区支持回跳到对应模块
- 输入聚焦时已经考虑移动端可视区域问题
- CTA 也已有防重复点击和提交中态

结论：

- 不再把“补基础交互态”作为下一轮核心任务
- 后续只需要围绕新功能继续补细节，不建议单独再做一轮“纯交互润色”

## 5. 候补或改约表达

- `v2` 已经做了非常轻量的异常兜底，例如：
  - 已满时段不可选
  - 紧张时段下对大桌提交做最终拦截
- 但页面还没有把“不可订之后怎么办”说清楚
- 用户看到 `已满` 或被拦截后，仍需要自己重新扫描替代选项

结论：

- 这一项不能算完成
- 它会成为本轮新版 P2 的核心任务之一

---

## 二、P2 阶段现在真正要解决什么

结合 `v2` 页面现状，这一轮 P2 重点不再是“页面像不像预订页”，而是进一步解决以下四个问题：

1. 当用户选到已满或紧张时段时，页面能不能主动给出替代路径
2. 人数和特殊需求，能不能真正影响推荐逻辑，而不是只停留在提示文案层
3. 用户提交后，能不能感受到更完整的后续状态和可操作空间
4. 页面里的各种状态文案，能不能形成一套更统一、更可信的表达体系

这轮 P2 的目标不是继续堆内容，而是让页面从“高完成度原型”更进一步接近“真实产品页的决策体验”。

---

## 三、本轮 P2 优化范围

本阶段建议重点处理：

- 不可订场景的替代路径
- 人数与需求驱动的推荐逻辑
- 提交后的状态与后续操作
- 状态表达一致性

本阶段不建议重复处理：

- 再补一轮基础库存标签
- 再补一轮基础成功 toast
- 再做一遍人数快捷选择
- 单纯再做一轮视觉翻新

这些内容在 `v2` 中已经有了较好的基础，继续投入的边际收益已经明显下降。

---

## 四、本轮 P2 具体优化指导

## 1. 从“不可选”升级到“给出替代路径”

## 当前问题

- `v2` 对已满时段的处理仍以禁用为主
- 用户能看到某个时段不可订，但页面没有主动告诉他“那你现在最适合改选什么”
- 紧张时段的大桌兜底拦截发生在提交时，仍然偏晚

## 优化目标

把“不能订”从一个死胡同，变成一个可继续推进的分岔口。

## 指导要求

- 当用户选中某一天后，如果晚市大量时段紧张或已满，应在时间区域附近给出轻量替代建议，例如：
  - 同一天更稳妥的前置时段
  - 邻近日期的可订晚市时段
  - 更适合大桌的日期提示
- 如果要做候补表达，建议只做到轻量级：
  - `已满可候补`
  - `如该时段释放库存会优先联系`
  - `先提交候补需求，门店二次确认`
- 不建议把候补扩展成完整流程系统，例如：
  - 候补队列排序
  - 候补通知中心
  - 独立候补订单页
- 替代建议必须基于页面中真实存在的日期和时段组合，不要凭空写一个推荐时间
- 大桌场景下，替代建议应更保守，例如优先引导用户改到：
  - 更早时段
  - 非高峰日
  - 明确标注为更适合聚餐的人数区间

## 交付标准

- 页面不再只会说“当前不可预订”
- 用户在不可订场景下能立即看到下一步建议，而不是自己重新全文扫描

---

## 2. 让人数与需求真正参与推荐逻辑

## 当前问题

- `v2` 已经能根据人数变化提示文案
- 但时间状态和推荐逻辑基本仍是静态的
- 备注标签目前主要进入确认区，没有真正影响页面建议

## 优化目标

让“2 人”“6 人以上”“包间需求”“安静位置”“儿童座椅”等信息，不只是被记录，而是真正参与决策。

## 指导要求

- 当人数较多时，应对部分时段给出更明确的风险提示，例如：
  - `该时段对 6 人以上较紧张`
  - `大桌建议改选 17:30 或次日 18:00`
- 当用户选择了 `包间`、`儿童座椅`、`安静位置` 等标签后，可在时间区或确认区追加一层轻提示，例如：
  - `包间需求建议尽量避开晚市高峰`
  - `带娃场景可优先选择午市或较早晚市`
  - `安静位置更建议非高峰档`
- 推荐逻辑可以是“建议层”的变化，不要求真的改写库存数据
- 不要因为用户选了某个标签，就直接伪造“该时段支持包间”这类强承诺
- 建议统一使用：
  - `更稳妥`
  - `更容易协调`
  - `建议优先`
  这类保守表达，避免像后端库存系统一样给出确定性结论

## 交付标准

- 用户会感受到页面在“根据我的场景给建议”
- 推荐不再只是对所有人都一样的静态标签

---

## 3. 提交后从“成功提醒”升级到“后续状态卡片”

## 当前问题

- `v2` 的成功卡片已经能告诉用户“提交成功”
- 但提交后仍然偏一次性提示
- 用户看完成功卡片后，不知道还有没有别的动作可以做

## 优化目标

让提交后的页面不仅有结果，还能表达接下来会发生什么，以及用户还能做什么。

## 指导要求

- 成功卡片中建议增加更清晰的流程表达，例如：
  - `已提交预订申请`
  - `门店确认中`
  - `短信通知结果`
- 可增加 1 到 2 个轻操作入口，例如：
  - `返回修改时间`
  - `查看定金规则`
  - `拨打门店电话`
- 如果用户提交的是紧张时段，可在成功态中补一句更真实的说明，例如：
  - `若该时段无法确认，门店可能优先与您协商临近时段`
- 当前页面里类似“这是原型演示页”的说明，应尽量弱化或下沉，不要让用户成功后第一眼看到的还是设计解释
- 不需要接订单中心，也不需要扩展成完整“我的预订”体系

## 交付标准

- 提交后不再只是短暂被通知一下
- 页面能继续承接用户对“然后呢”的心理预期

---

## 4. 补强状态表达的一致性与可信度

## 当前问题

- `v2` 已经有很多状态词，但这些状态的解释还比较分散
- 用户知道某个时段是 `推荐` 或 `紧张`，但不一定知道“为什么”
- 首屏说明、日期提示、时间标签、提交前说明之间还可以更统一

## 优化目标

让整页状态表达形成一套更完整的语言系统，减少“写了很多状态词，但理由不够清楚”的感觉。

## 指导要求

- 为核心状态建立更稳定的含义边界，例如：
  - `可订`：当前可正常提交
  - `推荐`：当前更稳妥或体验更好
  - `紧张`：可尝试提交，但确认存在不确定性
  - `已满`：当前不可直接预订
- 推荐态最好能补一层理由，不一定每个时段都写，但至少在页面某处让用户理解：
  - 为什么推荐
  - 推荐对什么场景更合适
- 日期提示、时间状态、确认区标签、提交成功说明，尽量使用同一套词汇体系
- 如果页面继续使用趋势型表达，就不要突然混入过于精确的数字库存
- 如果当前选择的是紧张时段，确认区和提交说明也应同步体现风险，而不是只在时间卡片上出现一次

## 交付标准

- 页面里的状态词不再像零散文案，而像一套有规则的产品表达
- 用户可以理解这些状态的业务含义，而不是只看颜色和标签

---

## 5. 让“回改”能力从提交前延伸到提交后

## 当前问题

- `v2` 的吸底确认区已经支持回跳修改
- 但这种能力主要发生在提交前
- 一旦进入成功状态，用户缺少继续改约或重选的明显入口

## 优化目标

让“先提交再回改”的路径也更自然，增强原型的真实感。

## 指导要求

- 成功卡片中可提供返回修改入口，直接滚动到：
  - 日期区
  - 时间区
  - 联系信息区
- 如果用户重新修改已提交过的信息，可将成功卡片降级为“待重新提交”的状态，而不是简单保留旧成功文案
- 不需要实现真实订单覆盖逻辑，但要在原型层把“修改后需要重新提交确认”表达清楚
- 对于这类交互，优先做轻量状态切换，不建议扩展成复杂的多步骤流程

## 交付标准

- 页面不再只有“提交前能改、提交后不能动”的单向心智
- 用户更容易理解这是一个可回退、可调整的真实预订过程

---

## 五、P2 阶段完成标志

当页面达到以下状态时，可以认为这一轮新版 P2 基本完成：

1. 已满或紧张场景下，页面能主动提供替代路径
2. 人数和特殊需求开始影响推荐建议，而不只是写进备注
3. 提交后具备更完整的后续状态表达和轻操作入口
4. 各种状态标签形成更统一的表达体系
5. 修改路径可以从提交前延伸到提交后

---

## 六、P2 阶段边界提醒

以下内容仍然不建议在本题继续扩展：

- 真实支付接口
- 登录注册体系
- 后端库存系统
- 真实短信发送能力
- 订单中心与完整改签系统
- 复杂优惠券、会员、积分模块

这些能力会迅速把题目从“预订体验优化”拉到“完整业务系统搭建”，不适合当前题目体量。

---

## 七、本轮 P2 一句话原则

新版 P2 不是继续给页面堆更多模块，而是让已有状态真正驱动决策、替代路径和后续操作，让页面更像一个能把用户带到下一步的真实预订产品。
